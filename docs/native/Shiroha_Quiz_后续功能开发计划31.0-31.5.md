# Shiroha Quiz 原生 Android / Compose 版 31.0～31.5 开发计划

> 说明：本文是阶段性规划。当前项目状态请优先参考根目录 `README.md`、`CHANGELOG.md`。
> 30.5～31 阶段（刷题体验、批量做题、偏好记忆）已在 v0.4.x 基本交付。

## 一、阶段目标

31.0～31.5 阶段围绕 **导入质量、AI 辅助、数据持久化、工程规范** 展开。

本阶段重点从刷题体验转移到工具链和工程质量：

```text
深化导入校验与修正能力
完善 AI 核对 / 解析的实用性
迁移到结构化持久化方案（Room）
补齐自动化测试和 CI
```

整体策略：

```text
先做导入和 AI 的体验深挖（用户感知强）
再做持久化和性能（内部收益大）
最后做测试和 CI（工程保障）
```

---

## 二、开发范围与原则

本阶段默认只处理原生 Android / Compose 版：

```text
apps/android/app/src/native/
```

除非明确要求，不改：

```text
apps/web/
apps/android/app/src/main/assets/web/
packages/
```

继续坚持：

1. 默认交付最小替换包
2. 只打包实际修改文件
3. 不回退前面已确认功能
4. 素材只使用 WebP 格式（q70）

# 三、31.0 计划：导入质量深挖

## 1. 目标

31.0 在 v0.4.x 导入能力基础上，进一步深挖校验和修正体验。

重点解决：

```text
导入预览中的批量修正仍不够方便
异常题目的修复建议不够具体
大题库（500+ 题）导入预览卡顿
Excel 表格中混合布局识别不稳定
```

## 2. 批量修正增强

### 2.1 功能目标

在导入预览中提供更高效的批量操作。

建议增加：

```text
按题型批量修改答案格式
按分区统一设置题型
异常题目一键应用 AI 建议
选中多题后统一操作
```

### 2.2 交互建议

```text
长按进入多选模式
顶部显示已选数量
底部批量操作栏（改题型 / 删 / 应用建议）
```

---

## 3. 导入预览性能优化

### 3.1 功能目标

当前 500+ 题导入预览渲染卡顿。建议：

```text
LazyColumn 替代一次性全量渲染
首屏只渲染可见区域 + 前后预加载
筛选计算延迟到后台线程
```

### 3.2 验收标准

```text
500 题导入预览无明显卡顿
筛选切换 < 200ms
滚动帧率 ≥ 30fps
```

---

## 4. Excel 表格解析增强

### 4.1 功能目标

当前 Excel 解析对混合布局（单列多题、合并单元格）支持有限。

建议增加：

```text
自动检测表头行和列映射
支持合并单元格识别
多 Sheet 自动选择
```

---

## 5. 31.0 验收标准

```text
批量修正功能可用（多选 + 统一操作）
导入预览 500 题无明显卡顿
Excel 混合布局识别率提升
原有导入功能不回退
```

# 四、31.1 计划：AI 辅助增强

## 1. 目标

31.1 提升 AI 核对和 AI 解析的实际可用性。

重点解决：

```text
AI 核对结果展示不够清晰
AI 解析生成后缺乏二次确认
用户不清楚 AI 做了什么、改了哪里
```

## 2. AI 核对 diff 视图

### 2.1 功能目标

将 AI 核对结果以 diff 形式展示，用户能直观看到修改点。

展示建议：

```text
原始答案 → AI 建议答案
题型变更标记
解析补充标注
```

### 2.2 交互建议

```text
逐条确认 / 拒绝
支持一键接受全部建议
接受前预览变更影响
```

---

## 3. AI 解析质量反馈

### 3.1 功能目标

让 AI 解析的生成过程更加透明。

建议：

```text
显示解析生成耗时
展示 AI 模型的推理过程（可选）
解析质量自评（AI 对自己生成内容的信心度）
```

---

## 4. 31.1 验收标准

```text
AI 核对结果以 diff 对比展示
可逐条接受 / 拒绝建议
一键接受全部可用
AI 解析带质量信心提示
原有 AI 功能不回退
```

# 五、31.2 计划：数据持久化升级

## 1. 目标

当前 SharedPreferences + JSON 方案在题库变大时出现明显瓶颈（导入慢、启动慢、内存高）。

迁移到 Room（SQLite）持久化。

## 2. 迁移策略

```text
先做 Room 基础表（Bank / Question / Record / WrongBook）
保留 SharedPreferences 读兼容
首次启动自动迁移现有数据
双写过渡期后移除旧方案
```

## 3. 预期收益

| 指标 | 当前（SP JSON） | 目标（Room） |
|------|----------------|-------------|
| 启动加载 | 全量反序列化 | 按需查询 |
| 题库切换 | O(n) 序列化 | O(1) 引用 |
| 导入 500 题 | ~3s | <1s |
| 内存占用 | 全量驻留 | 分页加载 |

## 4. 数据迁移注意事项

```text
迁移过程不能丢失现有用户数据
迁移失败要有明确提示和回滚
迁移后 schema 版本号更新
备份 JSON/ZIP 导出格式不变（仅导入时映射到 Room）
```

## 5. 31.2 验收标准

```text
Room 数据库完整替代 SP JSON
现有题库数据自动迁移无丢失
导入 / 练习 / 考试 / 记录流程不变
启动速度明显提升
内存占用降低
备份 JSON/ZIP 导出格式不受影响
```

# 六、31.3 计划：练习体验增强

## 1. 目标

基于 30.5～31 阶段已交付的刷题功能，继续做细节打磨。

## 2. 填空题答案归一化

### 2.1 功能目标

当前填空题仅支持 `||` 分隔等价答案。扩展为：

```text
支持通配符或正则匹配（如 "XXX 年" 匹配任意年份）
支持数字范围（如 "10～15" 匹配 10-15 之间）
忽略标点符号差异
忽略空格差异
全角/半角自动对齐
```

### 2.2 题型检测优化

需要判断“短填空答案” vs “短简答答案”：
- 填空题通常是客观、唯一的（人名、地名、数字、定义）
- 简答题通常是主观、开放的（理由、说明、方法）

## 3. 简答题手动评分

考试模式下，简答题由用户自行评分：

```text
交卷后简答题显示参考答案
用户自行判断得分（0/部分/满分）
得分计入最终成绩
```

## 4. 收藏 / 标记功能

```text
练习和考试中可收藏题目
错题本中可标记掌握状态（未掌握 / 已掌握 / 需复习）
收藏题目可在独立列表查看
```

## 5. 31.3 验收标准

```text
填空题归一化支持通配符/正则
填空题归一化不影响现有判断逻辑
简答题手动评分可用
收藏 / 标记功能完整
原有刷题流程不被破坏
```

# 七、31.4 计划：测试与 CI

## 1. 目标

补齐自动化测试和持续集成。

## 2. 测试范围

### 2.1 单元测试（核心）

```text
解析器链各步骤的单元测试（TextImportDecoder / StandardQuestionParser / AnswerParser）
答案匹配逻辑测试（labeled answer / 集中答案区 / 括号答案）
题型推断测试（填空 vs 简答边界）
图片处理测试（base64 解码 / WebP 压缩）
数据迁移测试（SP JSON → Room）
```

### 2.2 集成测试

```text
完整导入链路（文本 → 解析 → 校验 → 导入）
练习状态流程
考试倒计时与自动交卷
备份导入导出
```

### 2.3 覆盖率目标

```text
解析器核心：≥ 80%
状态管理：≥ 70%
UI 组件：≥ 50%（优先解析器）
```

## 3. CI 配置

```text
GitHub Actions
  └── PR / Push 触发
      ├── ./gradlew assembleNativeRelease（编译检查）
      ├── ./gradlew testNativeReleaseUnitTest（单元测试）
      └── 产出 APK 供手动测试
```

## 4. 31.4 验收标准

```text
解析器核心模块测试覆盖率 ≥ 80%
CI 自动编译通过
CI 自动运行测试
PR 合并前 CI 必须通过
```
