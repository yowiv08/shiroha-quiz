# Shiroha Quiz 后续功能开发计划 33-34

当前版本：原生 `v0.8.1-native`，Web `v0.7.0-alpha`。原生 Compose 版为主开发线，Web 端同步演进解析架构。

---

## 一、当前阶段定位

33 阶段结束后，原生 Compose 版已覆盖完整的导入→练习→考试→错题→复习→AI 辅助闭环，Web 端解析架构从 V33 升级至 V34。33-34 阶段的核心任务是：解析稳定性收口、跨端数据互通、分组练习范围、路由系统升级、为后台化接入做铺垫。

当前核心链路已具备：

- 题库导入（docx/txt/xlsx/json，6 种 JSON 格式）、多策略解析、导入预览、核对、写入；
- 普通练习、即时判题、自动下一题、批量练习；
- 背题模式、斩题功能、题目收藏、快速编辑；
- 选项打乱（练习+考试独立控制），固定 A/B/C/D 标签仅随机内容映射；
- 考试、计时、交卷、记录明细、逐题复盘；
- 错题本、连续掌握规则、作用域切换（当前题库/全部题库）；
- 智能复习模式、首页待复习提示；
- AI 核对、AI 解析、AI 单题分析、编辑器 AI 补解析；
- 字号缩放、紧凑选项、顺序练习进度记忆；
- 首页紧凑布局、显示异常设备兼容；
- 双图标系统、Shiroha 模式（开屏/插画/图标开关）；
- 素材 WebP 化、暗夜/浅色模式、Design Token 系统；
- Excel 表格导入、JSON 跨端互导、图片题互通（ZIP + base64）；
- 导入编辑器查找替换（正则）；
- 主观题防误判、英文缩写防误判、回归测试 21/21；
- R8 + 资源收缩 + 正式签名；
- 跨端数据互通（错题本/收藏夹/记录跨端互导）。

因此 33-34 阶段不建议大范围重构，而应优先围绕三个方向推进：

1. **解析稳定性收口**：修复已知回归，扩展回归保护。
2. **跨端数据互通**：错题本/收藏夹/记录实现跨端互导。
3. **工程与后台化铺垫**：为大题库场景、CI 和 Android 接入后台打好基础。

## 二、33 阶段前已完成能力清单

以下能力在 33 阶段前已经落地，后续阶段不得回退：

### 1. 视觉与主题

- 双图标系统统一至 native 源集；
- Shiroha 模式（开屏图 / 插画 / 图标统一开关）；
- 素材全面 WebP + 暗夜模式无边框；
- 暗夜 / 浅色模式切换 + Design Token 系统。

### 2. 导入链路

- 题库 JSON 相互导入，含图片完全互通（ZIP + base64）；
- 导入编辑器查找替换（正则）；
- 填空题识别优化（关键词扩展，减少简答题误判）；
- 题干中包含"格式"时的分区误识别修复；
- 核对页筛选器按需显示 + 自动聚焦；
- 单题编辑页 + AI 建议前置；
- Excel 表格导入 + 数学 / 化学符号识别；
- JSON 导入 6 种格式全面兼容（多题库数组/题目数组/单题库对象/Web 备份包/对象选项）；
- 答案解析多别名兼容（answer/answers/correctAnswer/correctAnswers）。

### 3. 练习与考试

- 练习 / 考试主流程稳定（答题、着色、收卡、滑动切题、答题卡）；
- 背题模式、斩题功能、题目收藏和题目快速编辑；
- 智能复习模式、错题掌握状态和首页待复习数量提示；
- 字号缩放、紧凑选项、顺序练习进度记忆，退出练习时可保存当前位置；
- 选项打乱（练习+考试独立控制）；
- AI 单题分析（练习提交后/背题模式触发）；
- 编辑器 AI 补解析（四个入口统一集成）。

### 4. 错题与记录

- 错题本作用域切换（当前题库/全部题库）；
- 记录列表与详情"只看错题"筛选。

### 5. 兼容性

- 首页紧凑布局和显示异常设备兼容，题目文本允许自动换行；
- 二级页面系统返回映射，避免退回桌面。

### 6. 工程

- AI 核对与解析（OpenAI 兼容）；
- R8 + 资源收缩 + 正式签名；
- Question 云端 11 字段完整序列化。

## 三、33-34 阶段开发目标

### 目标 1：解析稳定性收口

优先处理：

- 修复紧凑多选答案提取（13 号回归）：`embeddedChoiceLetterPattern` 扩展支持 (A,B)/(A B)；
- 修复前言/注意事项过滤（17/18 号回归）：`pureFrontMatterLineRegex` + `isPureFrontMatterPseudoQuestion`；
- 扩展回归保护：新增英文缩写防误判用例（21 号），回归测试 12→21 用例；
- 继续核校真实题库导入结果，沉淀更多解析样例。

验收标准：

- 回归测试 21/21 全部通过；
- 标准 docx/txt/xlsx/json 解析不被破坏；
- 解析样例库持续积累。

### 目标 2：跨端数据互通

优先处理：

- Web 端导出适配 Native 格式：完整备份输出 `kind:shiroha_quiz`，wrongBook/favorites 对象→数组；
- Web 端导入兼容 Native 格式：识别 `kind` 字段，数组→对象映射；
- Native 端导入兼容 Web 格式：`crossPlatform` 包装解包，ISO 时间字符串解析，状态自动映射；
- 备份格式升级 v3：新增 `crossPlatformSchemaVersion`、`exportedBy` 等元数据。

验收标准：

- Web 端导出的完整备份可被 Native 端导入并恢复错题本/收藏夹/记录；
- Native 端导出的完整备份可被 Web 端导入并恢复相应数据；
- 仅导入题库、不包含状态的旧格式仍然兼容。

### 目标 3：分组练习范围

双端实现：

- 练习范围可从单个题库扩展为整个分组，跨题库出题；
- 统计和进度按分组维度聚合；
- sessionKey 重构，解决跨题库重复题的答案冲突；
- 分组徽标、练习范围信息条 UI。

验收标准：

- 选择分组后练习题目来自分组内全部题库；
- 进度和统计按分组维度正确聚合；
- 双端 UI 一致。

### 目标 4：路由系统升级（Native）

- `QuestionSearch`、`BankDetail` 子页面回退到 `BankList` 而非首页；
- `AppRouteSnapshot` 路由快照编码/解码；
- 回退栈支持多级导航。

### 目标 5：工程与后台化铺垫

优先处理：

- 大题库（500+ 题）导入性能摸底与优化；
- 解析器单元测试框架搭建；
- CI 集成回归测试自动化。

可选方向：

- Android 使用端登录界面 + 拉取云端任务（后台化 M8）；
- Android 使用端答题记录回传（后台化 M9）。

验收标准：

- 大题库导入不出现 ANR 或明显卡顿；
- 解析器核心路径有测试覆盖；
- CI 自动运行回归测试。

## 四、Web 端维护策略

Web 端当前定位：

- 解析架构已升级至 V34，与原生端共享核心解析规则；
- 继续作为在线体验入口（GitHub Pages）和 WebView 壳基础；
- 功能开发以原生版为主，Web 版只做解析兼容和跨端数据互通；
- 两端题库格式保持兼容，不做破坏性变更。

后续注意：

- 原生端新增字段时必须考虑 JSON 导出与 Web 端互认；
- Web 端功能迭代不影响原生端的核心开发节奏；
- 两端题库格式保持兼容，不做破坏性变更。

## 五、建议下一轮开发顺序

1. **回归修复** — 紧凑答案归一化、前言过滤、英文缩写保护，21/21 通过；

3. **大题库性能** — 500+ 题库导入性能；

5. **后台化 M8-M9** — Android 接入后台。

## 六、开发约束

后续开发必须遵守：

1. **默认开发范围** — 原生 `apps/android/app/src/native/`，Web `apps/web/`；
2. **最小修改原则** — 不顺手重构，不引入无关变更；
3. **不回退既有功能** — 已完成能力清单中的所有功能不得被破坏；
4. **标准解析优先** — 标准题库解析主线必须稳定，复杂兜底不能污染标准格式；
5. **后台化本地提交** — 后台模块（admin/server）只在 `content-admin` 分支本地提交，不推送远端；
6. **敏感信息隔离** — 代码、测试样例、文档中不得写入真实组织、人名等敏感信息；
7. **标签保护** — 永远不删除 Git 标签，只删除 Draft 草稿 Release。
