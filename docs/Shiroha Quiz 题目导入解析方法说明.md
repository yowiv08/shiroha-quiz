# Shiroha Quiz 题目导入解析方法说明

> 本文根据当前 Shiroha Quiz 源码中的导入与解析逻辑整理，主要说明软件在导入题库时是如何读取文件、识别题目、匹配答案、判断题型、生成预览和提示异常的。  
> 本文偏“解析原理说明”，适合使用者、维护者和后续开发者阅读。

---

## 一、总体解析流程

Shiroha Quiz 的题目导入不是简单按行切割，而是一个多阶段流程。

整体流程可以概括为：

```text
读取文件或文本
↓
提取纯文本
↓
判断导入类型
↓
执行题目解析策略
↓
识别题干、选项、答案、解析、题型
↓
进行题目规范化
↓
生成识别结果预览
↓
检查异常和风险提示
↓
使用者确认后创建题库
```

在导入时，系统主要区分三类数据：

| 类型 | 处理方式 |
|---|---|
| 普通题库文本 | 进入题目解析流程 |
| 题目文件 + 答案文件 | 先分别解析，再匹配合并 |
| Shiroha Quiz 备份 JSON | 不走普通解析，直接走设置/导出页的备份恢复逻辑 |

---

## 二、文件读取方式

### 1. 普通文本文件

普通文本、CSV 等可以直接按 UTF-8 文本读取。

适合：

```text
TXT
CSV
复制粘贴文本
部分纯文本 JSON
```

### 2. Word 文档

当前主要支持 `.docx`。

`.docx` 的处理方式是：

```text
读取 docx 文件
↓
按 zip 结构解包
↓
找到 word/document.xml
↓
提取其中的文本节点
↓
转换为普通文本
```

注意：

> 旧版 `.doc` 暂不建议直接导入。  
> 如果题库是 `.doc`，建议先用 Word 或 WPS 另存为 `.docx`，再导入。

### 3. PDF 文件

PDF 会优先尝试使用本地 PDF.js 提取文字。

流程大致为：

```text
读取 PDF 二进制
↓
尝试使用 PDF.js 提取每页文字
↓
如果 PDF.js 不可用或失败，尝试轻量文本流提取器
↓
如果提取文字过少，提示可能是扫描版 / 图片版 PDF
```

注意：

> 当前主要适合文字型 PDF。  
> 扫描版 PDF、图片版 PDF、复杂排版 PDF 识别效果可能不稳定。  
> 这类文件建议先转成 Word / TXT，或复制其中可选中的文字再导入。

---

## 三、单文件智能导入解析方法

单文件智能导入适合题目、选项、答案在同一个文件中的题库。

### 1. JSON 结构化导入

如果文本以 `{` 或 `[` 开头，或者用户选择 JSON 模式，系统会优先尝试按 JSON 解析。

支持的常见结构包括：

```json
{
  "questions": []
}
```

或者：

```json
[
  {
    "question": "题干",
    "options": [],
    "answer": []
  }
]
```

JSON 导入成功后，会把每道题统一规范化为 Shiroha Quiz 内部题目结构。

### 2. 普通文本导入

如果不是 JSON，则进入文本题库解析流程。

文本解析会尝试识别：

- 题号
- 题干
- 选项
- 答案
- 解析
- 题型
- 分卷
- 分区
- 题组标题

---

## 四、文本题库的主要解析策略

Shiroha Quiz 的自动解析会综合多种策略，而不是只用一种规则。

### 1. 标准题块解析

系统会先把题库文本拆成多个题块。

常见切题依据包括：

```text
1. 题干
2、题干
第 3 题
一、单选题
二、多选题
判断题
第一卷
第二部分
```

每个题块内部再进一步解析：

```text
题干
选项
答案
解析
```

适合比较标准的题库格式。

---

### 2. 答案解析区解析

很多题库会把题目和答案解析分开，例如：

```text
一、单选题
1. 下列说法正确的是？
A. ...
B. ...

答案解析
1. 【答案】A
【解析】……
```

这类文件中，题目部分可能没有答案，答案集中出现在后面。

系统会尝试：

```text
先解析题目
↓
再识别答案解析区
↓
按题号和分区把答案合并回题目
```

支持的常见答案解析格式包括：

```text
1. 【答案】A
1.【答案】A【解析】……
答案解析 1. A
正确答案：B
参考答案：ABC
```

---

### 3. 分卷 / 分区解析

部分题库会分成多个卷、章节或题型区。

例如：

```text
第一卷
一、单选题
1. ...

二、多选题
1. ...

第二卷
一、判断题
1. ...
```

系统会尽量记录题目所在的：

- 卷
- 分区
- 题型标题

这样在答案合并时，可以减少“每个题型都从 1 重新编号”导致的错配。

---

### 4. 紧凑格式解析

有些题库格式非常紧凑，例如多个题目挤在一段里，或者选项和题干粘在一起。

系统会尝试做预处理：

```text
给题号前后补换行
给答案标记前后补边界
识别同一行中的 A/B/C/D 选项
拆分粘连的题干和选项
```

适合处理：

```text
1. 题干 A. 选项一 B. 选项二 C. 选项三 D. 选项四 答案：A
```

但如果原文过于混乱，仍然建议先人工或使用 LLM 智能体清洗格式。

---

### 5. 局部修复解析

如果整体解析大致成功，但少量题目存在异常，系统会尝试局部修复。

例如：

- 某一道题选项粘连
- 某一道题答案残留在题干里
- 某一道题被错误拆分
- 某几道题比整体格式更混乱

局部修复的原则是：

```text
只修少量风险题
不轻易替换整份解析结果
只有局部候选明显更好时才替换
```

这样可以避免“为了修一题，导致整份题库被重新错误解析”。

---

## 五、题块内部如何解析

每个题块内部会按以下方向识别。

### 1. 题号

支持常见题号形式：

```text
1.
1、
1．
第1题
1:
```

题号用于：

- 显示题目顺序
- 双文件答案匹配
- 答案解析区合并

### 2. 题干

题干一般是题号后的正文内容。

系统会尽量去掉：

- 答案标记
- 解析标记
- 多余编号
- 残留的选项前缀

### 3. 选项

支持常见选项形式：

```text
A. 选项
B. 选项
C. 选项
D. 选项
```

也支持一定程度的变体：

```text
A、选项
A：选项
A 选项
（A）选项
```

如果选项写在同一行中，系统会尝试拆分：

```text
A. 选项一 B. 选项二 C. 选项三 D. 选项四
```

### 4. 答案

支持常见答案标记：

```text
答案：A
正确答案：B
参考答案：ABC
标准答案：正确
Answer: A
Correct answer: B
```

支持答案形式包括：

| 答案形式 | 示例 |
|---|---|
| 单选字母 | A |
| 多选字母 | ABC |
| 分隔多选 | A、B、C |
| 数字答案 | 1、2、3 |
| 判断题答案 | 正确、错误、对、错、√、×、T、F |
| 文本答案 | 适合填空题、简答题 |

### 5. 解析

支持识别：

```text
解析：
【解析】
答案解析
```

解析内容会尽量保存到题目的解析字段中。

## 六、题型判断方法

题型判断会综合多个信息，而不是只看答案。

### 1. 优先看题型标题

如果题目前有题型分区，例如：

```text
单选题
多选题
判断题
填空题
简答题
```

系统会优先使用分区标题判断题型。

### 2. 根据选项判断

如果题目有 A/B/C/D 选项，一般会判断为选择题。

规则大致是：

| 情况 | 推断题型 |
|---|---|
| 有选项，答案只有一个 | 单选题 |
| 有选项，答案有多个 | 多选题 |
| 选项为“正确/错误”“对/错” | 判断题 |
| 题干含“多选”“多项选择” | 多选题 |

### 3. 根据题干判断

如果没有选项，则会看题干特征。

例如：

| 题干特征 | 推断题型 |
|---|---|
| 含“简答”“问答”“论述”“名词解释” | 简答题 |
| 含“填空”“补全”“____” | 填空题或简答题 |
| 答案是“正确/错误/√/×” | 判断题 |

### 4. 填空题与简答题的区别

如果没有选项，且答案是较长文本，系统更倾向于将其视为简答题。

如果题干有明显空格、横线、括号，且答案较短，系统更倾向于填空题。

---

## 七、答案规范化方法

系统会把不同格式的答案统一成内部结构。

### 1. 选择题答案

以下写法会尽量统一为选项 key：

```text
A
a
A、B、C
ABC
A B C
```

统一后类似：

```json
["A", "B", "C"]
```

### 2. 判断题答案

以下写法会统一映射：

```text
正确
对
是
√
T
True
```

对应“正确”。

```text
错误
错
否
×
X
F
False
```

对应“错误”。

如果判断题有 A/B 两个选项，系统会尝试把“正确/错误”映射到对应选项。

### 3. 文本答案

填空题和简答题会尽量保留文本答案，不强行拆成 A/B/C/D。

例如：

```text
参考答案：本地存储
```

会作为文本答案保存。

---

## 八、双文件导入解析方法

双文件导入适合题目文件和答案文件分开的情况。

### 1. 题目文件解析

题目文件会先按普通题库文本解析。

此时即使题目没有答案，也不会立即判定失败，因为答案可能在另一个文件中。

### 2. 答案文件解析

答案文件会尝试多种方式提取答案。

支持形式包括：

```text
1. A
2. ABC
3. 正确
```

也支持一行多个答案：

```text
1. A 2. B 3. C 4. D
```

还支持范围答案：

```text
1-5：A B C D A
```

如果答案文件本身也像完整题库，系统会尝试把它当作题库解析，再提取其中答案作为兜底。

### 3. 答案来源候选

双文件答案来源大致有四类候选：

| 候选方式 | 含义 |
|---|---|
| 答案表提取 | 直接从答案文件中提取题号和答案 |
| 答案解析区提取 | 从答案解析段中提取答案 |
| 完整题库解析兜底 | 把答案文件也当作完整题库解析，再提取答案 |
| 混合提取 | 综合以上多种候选，去重后使用 |

系统会给不同候选打分，优先采用质量更高的答案来源。

### 4. 题目与答案合并方式

合并时支持：

| 合并方式 | 说明 |
|---|---|
| 自动匹配 | 自动比较多种策略，选质量最高的一种 |
| 题型分组 + 题号匹配 | 适合每个题型都从 1 开始编号的题库 |
| 按题号匹配 | 适合整份题库题号唯一的情况 |
| 严格题号匹配 | 适合题号必须严格对应的情况 |
| 按顺序匹配 | 适合答案顺序与题目顺序完全一致的情况 |

自动匹配会综合考虑：

- 成功匹配到答案的题目数量
- 未匹配答案的题目数量
- 答案是否超出选项范围
- 判断题答案是否异常
- 合并后是否出现明显格式错误

---

## 九、识别质量评分

系统会对不同解析候选进行评分。

评分大致考虑：

| 指标 | 影响 |
|---|---|
| 识别出的题目数量 | 题目越多，基础分越高 |
| 是否缺少答案 | 会降低评分 |
| 是否缺少选项 | 会降低评分 |
| 答案是否超出选项范围 | 会明显降低评分 |
| 题干是否残留答案标记 | 会降低评分 |
| 选项是否疑似粘连 | 会降低评分 |
| 题目长度是否异常 | 会降低评分 |
| 是否符合预估题量 | 越接近越好 |

最终系统会选择综合得分更高的解析结果。

---

## 十、识别结果预览与异常提示

解析完成后，系统会进入识别结果预览。

预览不是装饰功能，而是导入流程中的重要检查环节。

常见异常包括：

| 异常 | 含义 |
|---|---|
| 缺少题干 | 没有识别到有效题目内容 |
| 选择题缺少选项 | 题型是单选/多选，但没有识别出 A/B/C/D |
| 缺少答案 | 没有识别到参考答案 |
| 答案超出选项范围 | 例如只有 A/B/C，却识别到 D |
| 单选题出现多个答案 | 单选题答案被识别成多个 |
| 多选题只有一个答案 | 多选题可能被误判，或答案不完整 |
| 判断题答案无法映射 | 正确/错误无法对应到选项 |
| 题干残留答案标记 | 题干中仍然有“答案：A”等内容 |
| 选项疑似粘连 | 多道题或答案解析可能粘到选项里 |

建议使用者在预览中重点查看：

```text
题型
题干
选项
答案
状态
异常提示
```

确认无误后再创建题库。

## 十一、备份 JSON 与普通题库 JSON 的区别

Shiroha Quiz 中有两类 JSON：

### 1. 普通题库 JSON

普通题库 JSON 通常只包含题目，例如：

```json
{
  "questions": []
}
```

或者：

```json
[
  {
    "question": "题干",
    "answer": "A"
  }
]
```

这类可以作为题库导入。

### 2. 备份 JSON / 批量题库 JSON

Shiroha Quiz 导出的完整备份通常包含：

```json
{
  "app": "Shiroha Quiz",
  "schemaVersion": 1,
  "banks": [],
  "wrongBook": {},
  "records": [],
  "settings": {}
}
```

这类 JSON 不应当走普通题库解析，而应进入：

```text
设置/导出 → 导入配置 / 备份 JSON
```

## 十二、复杂题库的处理建议

如果原题库非常复杂，建议先清洗，再导入。

尤其是以下情况：

- 题号不统一
- 选项有时在同一行，有时分多行
- 答案有时写在题后，有时集中在最后
- 题型标题混乱
- PDF 复制出来有大量换行错乱
- 题目中混有页眉、页脚、目录、说明文字

可以优先使用 LLM 智能体进行数据清洗，例如：

```text
豆包
深度求索
通义千问
```

清洗目标不是改题，而是统一格式。

可以要求 LLM 按下面格式输出：

```text
1. 题干
A. 选项一
B. 选项二
C. 选项三
D. 选项四
答案：A
解析：……
```

如果是多选题：

```text
2. 题干
A. 选项一
B. 选项二
C. 选项三
D. 选项四
答案：ABC
解析：……
```

如果是判断题：

```text
3. 题干
答案：正确
解析：……
```

清洗后再导入 Shiroha Quiz，识别准确率通常会更高。

## 十三、当前解析能力边界

当前解析能力较强，但仍有边界。

### 较适合

- 结构清楚的 TXT / Word
- 文字型 PDF
- 题号清晰的题库
- 选项格式相对统一的题库
- 答案格式相对统一的题库
- 题目和答案分离但题号清楚的题库

### 不太适合

- 扫描版 PDF
- 图片题为主的题库
- 题干大量依赖表格和图片的题库
- 题号混乱且答案也无序的题库
- 选项、答案、解析严重粘连的题库
- OCR 错字很多的题库

遇到这类题库时，建议先做文本整理或格式清洗。

---

## 十四、简要结论

Shiroha Quiz 的题目导入解析可以理解为：

```text
不是只按固定格式导入，
而是先读取文本，
再使用多种解析策略生成候选结果，
然后根据题目数量、答案完整性、异常情况进行评分，
最后选择质量较高的结果供使用者预览确认。
```

使用时最重要的原则是：

```text
普通题库 → 导入题库页
题目答案分开 → 双文件导入
软件备份 JSON → 设置/导出页
识别完成后 → 先看预览，再确认导入
题库过于混乱 → 先用人工或 LLM 智能体清洗格式
```
