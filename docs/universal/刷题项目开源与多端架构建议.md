# 刷题项目开源与多端架构建议

## 1. 总体结论

建议将网页版、后续安卓 App、共享刷题逻辑、题库解析逻辑和项目文档放在**同一个 GitHub 仓库**中，但不要把所有代码混在一起，而是采用 **monorepo（单仓库多项目）结构**。

推荐路线是：

> 一个 GitHub 仓库 + 清晰的 monorepo 目录结构 + 当前主攻 Web 端 + 提前抽离核心逻辑 + 后续再接入安卓 App 或移动端版本。

也就是说：**同一个仓库，多个清晰子项目。**

---

## 2. 为什么建议放在同一个仓库

### 2.1 项目核心不是“网页”，而是“刷题逻辑”

这个项目的长期核心能力包括：

- 题库导入；
- 选择题、判断题、填空题等题型识别；
- 标准题与非标准题的解析策略；
- 答案判断；
- 错题本；
- 刷题统计；
- 题库导入导出；
- 后续可能的多端数据同步。

这些逻辑不应该只写在 Web 页面里。否则未来开发安卓 App 时，很容易出现：

- 网页版和 App 版题目识别规则不一致；
- 判断题策略两边实现不同；
- Web 端修了 bug，App 端忘记同步；
- 题库格式在两个项目里分叉。

所以，刷题核心逻辑应该沉淀为共享模块。

---

### 2.2 开源早期不适合拆太多仓库

如果一开始拆成多个仓库，例如：

```txt
question-web/
question-android/
question-server/
question-core/
question-docs/
```

看起来边界很清楚，但实际会带来几个问题：

- 新用户不知道应该先看哪个仓库；
- README、Issue、路线图分散；
- 版本管理麻烦；
- 多端联动修改需要跨仓库提交；
- 开源贡献者上手成本更高。

对早期项目来说，集中在一个仓库更容易维护，也更利于展示项目整体形态。

---

### 2.3 monorepo 适合多端项目

monorepo 的价值不是“所有东西堆在一起”，而是让多个项目共享代码、共享类型、共享配置、共享文档，同时仍然保持目录边界。

适合你的原因是：

- Web 端和 App 端可以共用题目数据结构；
- Web 端和 App 端可以共用题库解析规则；
- 后端接口协议可以与前端类型保持一致；
- 文档、图标、设计规范可以集中维护；
- 以后开源协作时，Issue 和 PR 都在一个地方。

---

## 3. 推荐项目结构

建议采用下面这种结构：

```txt
question-practice-app/
├─ apps/
│  ├─ web/              # 网页版，当前主要开发的电脑端/网页端
│  ├─ android/          # 未来原生安卓 App，如果使用 Kotlin / Java
│  └─ mobile/           # 如果未来使用 React Native / Expo，可放这里
│
├─ packages/
│  ├─ core/             # 刷题核心逻辑：答题、计分、错题、记录
│  ├─ parser/           # 题库导入、题型识别、选项解析、智能判断策略
│  ├─ types/            # 题目、题库、答题记录等通用类型
│  ├─ ui/               # 可复用 UI 组件，主要给 Web / React Native 用
│  └─ shared/           # 通用工具函数
│
├─ server/              # 如果后续有用户系统、云同步、云题库等后端功能
├─ docs/                # 使用说明、开发文档、路线图、截图
├─ assets/              # Logo、图标、演示图片、设计素材
├─ .github/
│  ├─ workflows/        # GitHub Actions、自动构建、自动测试
│  └─ ISSUE_TEMPLATE/   # Issue 模板
│
├─ README.md
├─ LICENSE
├─ CONTRIBUTING.md
├─ CHANGELOG.md
├─ package.json
├─ pnpm-workspace.yaml  # 如果使用 pnpm workspace
└─ .gitignore
```

## 4. 当前阶段建议怎么做

### 第一阶段：先做好 Web 端

当前重点仍然应该是电脑端/网页版。建议先保持轻量结构：

```txt
apps/web/
packages/core/
packages/parser/
packages/types/
docs/
assets/
```

这个阶段不建议一上来就引入过重的工程工具。可以先用：

- npm workspaces；
- pnpm workspaces；
- yarn workspaces。

如果你已经习惯 pnpm，可以优先使用 pnpm workspace。

---

### 第二阶段：抽离核心逻辑

要重点避免把所有逻辑都写在页面组件里。建议把下面这些能力从 Web 页面中抽出来：

```txt
packages/core/
├─ answer-checker.ts       # 答案判断
├─ question-model.ts       # 题目结构
├─ score.ts                # 计分逻辑
├─ wrong-book.ts           # 错题逻辑
├─ practice-session.ts     # 一次刷题会话

packages/parser/
├─ import-text.ts          # 文本导入
├─ detect-question-type.ts # 题型识别
├─ normalize-options.ts    # 选项标准化
├─ judgment-strategy.ts    # 判断题策略
├─ choice-strategy.ts      # 选择题策略
├─ fallback-strategy.ts    # 非标准题兜底策略

packages/types/
├─ question.ts             # Question 类型
├─ question-bank.ts        # QuestionBank 类型
├─ answer.ts               # Answer 类型
├─ record.ts               # 答题记录类型
```

你之前提到的策略，例如：

- 标准题走简单导入；
- 少量非标准题只对异常题启用复杂策略；
- 判断题增加“全部 / 题库 / 一半”等选项；
- 智能判断题量和题型；
- 避免因为一两道异常题让整个题库都走复杂解析；

这些都应该放在 `packages/parser` 或 `packages/core` 中，而不是直接写死在 `apps/web` 的页面代码里。

---

### 第三阶段：再决定 App 技术路线

后续安卓 App 有两条主要路线。

#### 路线 A：React Native / Expo

适合目标：

- 希望快速开发安卓 App；
- 希望尽可能复用 TypeScript 逻辑；
- 希望 Web 和 App 的业务逻辑保持一致；
- 不追求极限原生体验。

推荐结构：

```txt
apps/
├─ web/
└─ mobile/

packages/
├─ core/
├─ parser/
├─ types/
└─ shared/
```

优点：

- 可以复用大量 TypeScript 逻辑；
- Web 和 App 的题库解析规则更容易一致；
- 后续维护成本较低；
- 比较适合个人开发和开源项目。

缺点：

- 原生能力需要通过 React Native / Expo 生态处理；
- 部分复杂原生功能可能需要额外适配。

---

#### 路线 B：原生 Android，Kotlin / Java

适合目标：

- 想做更纯粹的安卓原生 App；
- 熟悉 Android Studio；
- 未来希望深入使用原生能力；
- 不太在意 Web 和 App 共用同一套 UI 或业务代码。

推荐结构：

```txt
apps/
├─ web/
└─ android/

packages/
├─ schema/       # 题库 JSON 格式、接口协议、数据结构说明
├─ samples/      # 测试题库样例
└─ docs-core/    # 题库规则与解析规范
```

优点：

- 安卓体验更原生；
- 更适合复杂本地能力；
- Android Studio 支持更完整。

缺点：

- 很难直接复用 Web 的 TypeScript 业务逻辑；
- 主要复用的是题库格式、接口协议、设计资产和测试样例；
- 双端逻辑一致性需要额外测试保障。

---

## 5. 不推荐的结构

### 5.1 不推荐一开始拆多个仓库

不建议：

```txt
question-web/
question-android/
question-server/
question-docs/
```

原因：

- 开源展示割裂；
- 贡献门槛更高；
- 多端同步成本更高；
- 版本管理更复杂；
- 早期维护压力更大。

---

### 5.2 不推荐一个 src 目录塞所有内容

也不建议：

```txt
src/
├─ web 页面
├─ 安卓代码
├─ 后端代码
├─ 工具函数
├─ 图标
└─ 文档
```

这种结构表面上简单，但后期会迅速失控。尤其当项目同时存在 Web、App、后端、题库解析、图标素材时，必须提前划分边界。

---

## 6. 开源前需要准备的文件

建议至少准备：

```txt
README.md              # 项目介绍、截图、功能说明、运行方式
LICENSE                # 开源协议
CONTRIBUTING.md        # 贡献指南
CHANGELOG.md           # 更新记录
.env.example           # 示例环境变量，不能上传真实 key
SECURITY.md            # 安全说明，可选
CODE_OF_CONDUCT.md     # 社区行为规范，可选
```

## 7. 开源协议建议

### MIT

适合：

- 希望别人自由使用；
- 希望项目传播更快；
- 不介意别人基于你的项目做二次开发或商业使用。

特点：宽松、简单、开源项目常用。

---

### Apache-2.0

适合：

- 希望协议比 MIT 更完整；
- 关注专利授权条款；
- 希望项目更正式一些。

特点：宽松，但条款比 MIT 更细。

---

### GPL-3.0

适合：

- 希望别人修改后也必须继续开源；
- 不希望项目被闭源商业软件直接拿走。

特点：传染性更强，约束更严格。

---

### 对你的建议

如果这个刷题软件主要是个人开源项目，建议优先考虑：

```txt
MIT 或 Apache-2.0
```

如果你特别在意“别人改了也必须开源”，再考虑 GPL-3.0。

---

## 8. 图标和版权风险建议

因为你之前提到过二次元形象相关图标，所以开源时建议注意：

1. GitHub 开源也是公开传播，不等于没有版权风险；
2. 默认图标建议使用完全原创的通用刷题图标；
3. 与已有角色形象相关的版本不要作为默认项目品牌；
4. 可以把相关设计稿放到 `assets/concepts/` 或 `docs/design/`，并注明“概念稿 / 非默认品牌资产”；
5. README 首页、软件图标、Release 包里，最好只使用原创图标。

推荐资产结构：

```txt
assets/
├─ icons/
│  ├─ app-icon.svg          # 默认原创图标
│  ├─ app-icon.png
│  └─ app-icon-512.png
│
├─ concepts/
│  ├─ anime-inspired-v1.png # 概念稿，不作为默认品牌
│  └─ anime-inspired-v2.png
│
└─ screenshots/
   ├─ web-home.png
   ├─ import-question-bank.png
   └─ practice-page.png
```

## 9. 推荐的实施顺序

建议按照下面顺序推进：

```txt
1. 先整理当前 Web 项目目录
2. 新建 apps/web
3. 新建 packages/core
4. 新建 packages/parser
5. 新建 packages/types
6. 把题库导入、题型判断、答案判断逻辑从页面中抽出来
7. 补充 README.md
8. 补充 LICENSE
9. 补充 docs/architecture.md
10. 再考虑 apps/mobile 或 apps/android
```

---

## 10. 最终推荐方案

最终建议采用：

```txt
一个 GitHub 仓库
+ monorepo 目录结构
+ 当前主攻 apps/web
+ 抽离 packages/core 和 packages/parser
+ 后续再加 apps/mobile 或 apps/android
```

具体结构：

```txt
question-practice-app/
├─ apps/
│  ├─ web/
│  ├─ mobile/
│  └─ android/
│
├─ packages/
│  ├─ core/
│  ├─ parser/
│  ├─ types/
│  ├─ shared/
│  └─ ui/
│
├─ server/
├─ docs/
├─ assets/
├─ .github/
├─ README.md
├─ LICENSE
├─ CONTRIBUTING.md
├─ CHANGELOG.md
├─ package.json
└─ pnpm-workspace.yaml
```

一句话总结：

> 这个项目不应该拆成多个 GitHub 仓库，也不应该把所有内容塞进一个 src 目录。最合适的方案是一个仓库下的 monorepo：当前先做好 Web 端，同时提前抽离刷题核心逻辑，为后续安卓 App 和网页版长期共存打基础。
