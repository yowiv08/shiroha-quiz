# 架构说明

## 1. 总体思路

本项目采用单仓多项目结构，也就是常见的 `monorepo` 思路。

原因很简单：

- Web 和 Android 未来都会围绕“题库导入、刷题、考试、错题、记录”这些核心逻辑展开
- 如果一开始就拆成多个仓库，后期同步成本会更高
- 如果把所有内容混在一个目录里，后面边界会越来越乱

所以这里采用：

- 一个仓库
- 多个清晰子目录
- 后续把共享逻辑慢慢沉淀到 `packages`

## 2. 目录职责

### apps/web

当前通用网页版参考实现。

主要职责：

- 题库导入能力验证
- 导入策略验证
- 题库格式参考
- Web 侧功能演示

### apps/android

安卓原生应用主工程。

主要职责：

- 原生 UI 和交互
- 刷题体验
- 考试体验
- 本地记录与状态管理

### packages/core

后续沉淀核心刷题逻辑。

例如：

- 答案判断
- 计分
- 错题记录
- 刷题会话管理

### packages/parser

后续沉淀题库解析逻辑。

例如：

- 文本导入
- 题型识别
- 选项标准化
- 双文件合并

### packages/types

后续沉淀通用数据结构。

例如：

- Question
- QuestionBank
- AnswerRecord

### packages/shared

后续沉淀通用工具函数。

### packages/ui

后续沉淀可复用的 UI 规范或共享设计基础。

### docs

放产品、设计、页面、架构、执行计划等中文文档。

### assets

放图标、宣传图、设计素材。

## 3. 当前实现策略

当前阶段不追求一次把共享层全部抽出来，而是采用“边开发边抽离”的方式。

顺序是：

1. 先把产品结构和原生界面定下来
2. 再把真正重复使用的逻辑抽到 `packages`
3. 避免过早抽象

## 4. 当前安卓方向

安卓端当前采用过渡方案：

- WebView 壳（WebShellActivity 作为启动器）
- 加载 `apps/web/` 本地 web 资源
- 保留原生 MainActivity 作为后续原生开发入口

过渡期后计划逐步迁移至：

- Kotlin + Jetpack Compose
- 原生底部导航壳
- 原生页面组件体系

当前优先级：

1. 练习体验
2. 导入体验
3. 题库状态与本地模型

## 5. 首批页面

当前已经明确的核心页面有：

- 首页
- 练习页
- 导入页
- 我的页面

这几页已经有中文线框、组件说明和设计 Token 规范。

## 6. 后续推荐路线

推荐继续按照下面顺序推进：

1. 设计 Token 与组件体系
2. Compose 导航壳
3. 首页 / 练习 / 导入静态页面
4. 状态流
5. 题库模型
6. 导入逻辑
7. 考试模式
8. 错题与记录
